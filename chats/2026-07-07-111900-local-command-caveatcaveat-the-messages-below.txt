⏺ Good question to dig into. I read through parser.ts, intents.ts,
  dispatcher.ts, and voice.svelte.ts — the current setup is a wake word plus a
  regex/keyword parser over one Vosk transcript, with a scope stack and a 10 s
  pending-intent replay. That's a solid skeleton, and the "intelligence" gap is
  in three places: it only gets one shot at understanding you, it forgets
  everything after each command, and when it fails it can't repair. Here's what
  I'd do, roughly in order of payoff:

  1. Conversational memory — support follow-ups (biggest naturalness win, 
  cheapest to build).
  Right now "piano, show me D major" then "piano, now the minor" fails, because
  each utterance is parsed from scratch. Keep a small context slot (last root,
  quality/scale type, hand, inversion — essentially the last
  show-scale/show-chord intent) and let partial utterances patch it: bare
  "minor" → same root, minor; "what about F sharp" → same quality, new root;
  "left hand" already sort of works via set-hand, so this generalizes an
  existing pattern. This is pure parser/dispatcher work, fully unit-testable in
  vitest, no audio changes.

  2. Keep the conversation open after a command.
  The armed window (ARMED_WINDOW_MS) only opens after a bare "piano". After any
  successful command, re-arm for ~5 s so "faster… faster… stop" works without
  repeating the wake word. One-line concept in handleFinal; the main risk is
  piano notes decoding as commands during the window, but the grammar's [unk]
  handling plus restricting re-armed parsing to short/known phrases covers it.

  3. Repair instead of failing — "did you mean" and slot-filling.
  Two cases the dispatcher currently dead-ends on:
  - unknown intent → generic "Sorry, I didn't catch that." Instead, fuzzy-match
  the transcript against the active scopes' known phrases (you already have
  activePhrases() and a token-overlap scorer in matchLesson — same technique)
  and offer the best candidate: "Did you mean: show me D major?" Then accept
  "yes"/"no" while a confirmation is pending.
  - Partial intents → ask for the missing slot. "Show me the chord" with no root
  should get "Which root?" and treat the next single-note answer as the fill,
  rather than doing nothing. This is the same pending-intent machinery the
  dispatcher already has, pointed at the user instead of at a screen.

  4. Use more of what Vosk gives you (accuracy, not just UX).
  - N-best alternatives: if the WASM build supports SetMaxAlternatives (worth
  verifying — this build already rejected the grammar constructor, so treat it
  as maybe), parse each alternative and take the first that yields a known
  intent instead of only the top transcript. Homophone problems ("be flat" vs "b
  flat") largely dissolve.
  - Word confidence: with setWords(true) you get per-word confidence; act on
  high-confidence commands, confirm low-confidence ones ("D major — right?")
  instead of silently doing the wrong thing. Wrong-action-executed is much worse
  for trust than one extra confirmation.
  - Per-screen grammar: you rebuild one global grammar. Re-spawning the
  recognizer with a scope-narrowed word list when the route changes shrinks the
  search space and measurably improves recognition — though it costs a
  recognizer respawn per navigation, so worth profiling.

  5. Make it a coach, not just a remote control.
  This is the "actually intelligent" tier: voice that reads app state, not just
  writes it. Examples that fit your existing modules: "how am I doing" → speak
  matcher progress; "what should I practice" → pick from history.svelte.ts
  (least-recent or most-failed lesson); "what note is that" while the tuner
  listens; spoken corrections in lessons ("that was B natural — try B flat").
  All of this is dispatcher scopes + TTS, no new ML. One caveat from your own
  conventions: TTS strings must never contain "piano", and more talking means
  more time holding the playback gate that blinds the pitch detectors — coaching
  lines should stay terse.

  One option I'd not reach for first: routing unparsed utterances to an LLM for
  intent extraction. It would handle arbitrary phrasing, but it breaks the
  fully-on-device design, adds latency, and needs a key in a static-hosted
  browser app. Items 1–4 get you most of the perceived intelligence with zero
  network.

  If you want, I can start with 1+2 (context slots + re-arm window) — they're
  small, pure-parser changes with clear vitest coverage.

❯ What about an agentic tool? Is there a small model that has enough            
  performance to help with dispatching these voice commands?                    

⏺ Short answer: yes — this is actually a best-case task for a small model,
  because your intent space is tiny (~15 intent kinds, closed slot vocabularies)
  and Vosk's grammar already constrains what words can even appear in the
  transcript. But "small model" spans three very different tiers here, and I'd
  pick based on your on-device constraint rather than raw capability:

  Tier 1: Embedding-based intent matching (~25 MB, works everywhere) — the 
  sleeper pick.
  Not generative at all: run something like all-MiniLM-L6-v2 via transformers.js
  (quantized ONNX, ~23 MB, runs fine on WASM — no WebGPU needed). Embed a bank
  of example utterances per intent ("show me d major", "let me see the d major
  scale", "d major please"...), embed the incoming transcript, take nearest
  neighbor above a cosine threshold, then reuse your existing regex slot
  extractors (findRoot, findChordQuality, trailingNumber) to fill in the
  details. This handles phrasing variation — the thing your regexes are brittle
  at — at <50 ms latency, fully offline, and it's testable in vitest with fixed
  embeddings. It won't do conversational reasoning, but combined with the
  context-slot idea from before it covers most of the gap.

  Tier 2: A sub-1B instruct LLM in the browser via WebLLM (MLC) — genuinely 
  viable now.
  Models in the 0.5–1B class — Qwen3-0.6B, Llama 3.2 1B, Gemma 3 1B,
  SmolLM2-1.7B — are reliably good at "map this utterance to a JSON intent given
  a schema and 5 examples," especially with constrained JSON decoding, which
  WebLLM supports natively (it enforces your intent schema at the token level,
  so the model literally cannot emit an invalid intent). That eliminates the
  classic small-model failure mode of malformed output. Realistic costs:

  - Download: ~400–900 MB quantized (q4). Cacheable the same way you cache the
  Vosk model, but it's 10–20× your current largest asset.
  - Runtime: needs WebGPU. Chrome/Edge are fine; Safari's WebGPU support is
  recent enough that I'd treat it as progressive enhancement. On WASM/CPU
  fallback, latency becomes multi-second — not usable for voice.
  - Latency on WebGPU: prompt is short and output is ~30–50 tokens of JSON, so
  roughly 0.5–1 s on a mid-range machine. Acceptable for voice, noticeably
  slower than your current <1 ms parser.
  - Memory: you'd be running Kaldi WASM + tfjs WASM + an LLM simultaneously. On
  an 8 GB machine this is real pressure worth measuring before committing.

  The payoff over Tier 1 is actual dialogue intelligence: you can put the
  current screen, active scope phrases, and the last intent into the prompt, and
  the model handles "now the minor one", "same but left hand", "a bit faster"
  natively — no hand-coded context-slot logic.

  Tier 3: API model (Haiku) — smartest, but I'd rule it out for the core path:
  it breaks your offline/on-device design, adds network latency and a
  key-management problem in a static-hosted app. Only worth it as an explicit
  opt-in "smart mode."

  What I'd actually build: a tiered dispatcher. Keep the regex parser as the
  fast path — it's deterministic and instant for the 90% case ("piano, stop",
  "show me d major"). Only when it returns unknown (or low-confidence) does the
  utterance fall through to the model tier, along with the conversational
  context. That way the model's latency only shows up exactly where you
  currently show "Sorry, I didn't catch that" — pure upside. Start with Tier 1
  (small, universal, ~a day of work), and add the WebLLM tier behind a settings
  flag if the embedding approach still leaves gaps.

  One honest caveat: a chunk of your errors are upstream of parsing — the Vosk
  small model mishearing in the first place. An LLM can un-garble some of that
  ("be flat major" → Bb major) because it sees the whole utterance in context,
  and that recovery ability is arguably the strongest argument for Tier 2 over
  Tier 1.

  Want me to prototype the tiered dispatcher with the embedding approach? The
  intent-example bank and fallback wiring in dispatcher.ts are pure and
  unit-testable, so it slots into your existing test conventions cleanly.

     Embedding-Based Voice-Intent Fallback (Tier 1)

     Context

     The voice-control regex parser (src/lib/voice/parser.ts) only understands
     near-exact phrasings; anything else becomes {kind:'unknown'} → "Sorry, I
     didn't catch that." We're adding a Tier 1 fallback: when the parser returns
     unknown, embed the transcript with quantized all-MiniLM-L6-v2
     (transformers.js v3 on WASM, fully vendored/offline — no WebGPU, no
     network), nearest-neighbor match it against a bank of example phrasings per
     intent, and re-extract slots from the actual transcript. On any miss the
     original unknown intent is dispatched, so existing behavior is preserved.
     No generative LLM, no API calls (Tiers 2/3 explicitly out of scope). User
     emphasized: a test suite that genuinely challenges the work.

     Design principles carried from the codebase: matching logic stays pure
     (vitest-covered, no DOM/audio types); only voice.svelte.ts touches the
     impure embedder; dispatcher.ts is unchanged; model binaries are committed
     under public/ (precedent: 41 MB vosk tarball); transformers.js is
     dynamically imported like vosk-browser. False positives are worse than 
     misses — the app performs actions — so three stacked rejection gates:
     similarity threshold, cross-template margin, slot-resolution null.

  How it works: when the regex parser returns unknown, voice.svelte.ts embeds
  the transcript (quantized MiniLM via transformers.js on WASM, preloaded in the
  background when voice enables) and matches it against ~24 intent templates in
  intentBank.ts. The winning template re-extracts slots from the actual
  transcript — "give me a beat at eighty" matches the "give me a beat at ninety"
  example but resolves to bpm 80. Three rejection gates (similarity threshold
  0.55, cross-template margin 0.06, required-slot null) bias it toward "Sorry, I
  didn't catch that" over wrong actions. Fully offline: model and ort runtime
  are vendored in public/ like the Vosk model; a headless-browser check
  confirmed zero external requests. Cost when it fires: ~7 ms per query after a
  one-time ~1 s load.

  The test suite (221 new voice tests): pure matcher math with hand-built
  vectors; bank invariants (every template resolves its own examples, every
  example word provably in the Vosk grammar so paraphrases are decodable at
  all); resolver behavior with a deterministic fake embedder; and a real-model
  integration suite — 48 paraphrase rows each with an honesty guard proving the
  regex parser genuinely misses it, 9 adversarial negatives ("i like this
  piano", "what time is it") that must return null, and a calibration test
  pinning the measured separation (positives ≥ 0.57, negatives ≤ 0.50, min
  cross-template gap 0.11) so the constants can't drift silently.

  Things I learned by measuring, which changed the design:
  - Antonym intents (faster/slower, arpeggio/block) sit too close in embedding
  space for separate templates — each pair is now one template whose resolve()
  reads the direction word from the transcript.
  - I dropped the planned show-scale/show-chord/open-lesson templates: any
  transcript with a detectable root or lesson keyword is always consumed by the
  regex fast path first, so they were unreachable.
  - I fixed a real fast-path bug the work exposed: the homophone map made the
  verb "see" parse as the note C, so "piano, can I see the chords" used to show
  a C major scale. Homophones now require a following sharp/flat/quality
  keyword, with regression tests ("see sharp" still → C#).

  Two tradeoffs to know about: the repo gains ~58 MB of committed binaries (23
  MB model + 35 MB ort runtime — same convention as the 41 MB Vosk tarball), and
  the production build also emits a dead 23.5 MB wasm asset from
  transformers.js's internal URL reference (harmless dist bloat; documented in
  CLAUDE.md/memory). One thing I couldn't verify headless is the full mic → Vosk
  → fallback loop — worth one real "piano, can we pick up the pace" at the
  keyboard.
