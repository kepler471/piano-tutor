❯ do worktrees work off the local main or the remote

⏺ Off the local main — more precisely, off whatever commit the local checkout's
  HEAD was at when the worktree was created. Nothing is fetched from the remote.

  You can see it in this repo: this worktree's branch reflog says Created from 
  fa6fda3, which was your local main's tip when the session started. If
  origin/main had been ahead (or diverged) at that moment, the worktree would
  neither know nor care — it never consulted origin.

  Practical implications:

  - If your local main is stale, worktree sessions start from stale code. Do a
  git pull on main before spawning worktree sessions if you want them based on
  the latest remote state.
  - Merging back (like we just did) is also purely local — nothing has been
  pushed. origin/main on GitHub still doesn't have today's merge until you git 
  push.
  - All worktrees share one object store and branch namespace with the main
  checkout, so the moment you commit in a worktree, the main checkout can see
  that branch — no remote round-trip involved.
