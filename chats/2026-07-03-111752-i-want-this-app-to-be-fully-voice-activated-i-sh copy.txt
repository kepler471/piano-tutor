     Voice Control for Piano Tutor

     Context

     The user practices piano with both hands on the keys and no MIDI connection
     — the app already listens through the mic for pitch detection. They want
     to drive every app feature by voice so practice is fully hands-free:
     navigate screens, pick scales/chords, start lessons, control the metronome,
     play demos, start/stop listening, record, etc.

     User decisions (confirmed):
     - Engine: on-device Vosk WASM (vosk-browser + vosk-model-small-en-us-0.15,
     ~40 MB) — fully offline/private, fits the app's static-hosting model. Not
     the Web Speech API.
     - Activation: wake word ("piano, …"). Always listening while voice mode is
     on; anything without the wake prefix is ignored, which also filters
     piano-noise garbage transcripts.
     - Spoken confirmations via speechSynthesis.

     Key codebase facts driving the design:
     - No command layer exists: every screen holds selections as component-local
     $state mutated by inline onclick. Reusable module-level entry points
     today: navigate(route) (src/router.svelte.ts),
     playNote/playChord/playSequence (src/lib/audio/playback.ts),
     startMetronome/setMetronomeBpm/stopMetronome (metronome.ts),
     start/stopMonoDetection, start/stopPolyDetection, theory data (SCALE_TYPES,
     CHORD_ROOTS, CHORD_QUALITIES, inversionsFor), allLessons().
     - The mic is a shared singleton (src/lib/audio/mic.svelte.ts) exposing
     source/audioContext; polyPitch already taps it via an AudioWorklet
     (public/worklets/capture-processor.js). Voice will tap the same graph — no 
     second getUserMedia. Hazard: stopMonoDetection() calls mic.stop()
     unconditionally, so mic lifecycle needs refcounting once voice holds it
     open across screens.
     - playback.isPlaying (+300 ms tail) is the established "detectors, ignore
     the mic" gate; it is a module-internal binding, so TTS gating needs a small
     exported helper in playback.ts.
     - Conventions: reactive singletons are .svelte.ts modules with getters;
     parsing/detection logic stays pure for vitest (src/tests/); model assets
     are copied into public/ (existing public/model/, public/tfjs-wasm/
     pattern); localStorage keys piano-tutor.* with try/catch (history.svelte.ts
     pattern).
     - Reliability lever: Vosk's KaldiRecognizer accepts a grammar constraint
     (JSON list of allowed phrases + "[unk]") — generating it from the command
     phrase table makes the small model near-deterministic for commands and
     turns piano noise into [unk].

     Architecture

     mic.svelte.ts (shared getUserMedia + AudioContext — gains refcounting)
      └─ AudioWorkletNode 'capture-processor' (same worklet polyPitch uses)
          └─ Resampler(ctx.sampleRate → 16000)              (existing
     resample.ts)
              └─ vosk-browser KaldiRecognizer (Kaldi WASM in its own worker,
                 grammar-constrained to the command vocabulary + "[unk]")
                  ├─ partialresult → live transcript in VoiceHud
                  └─ result → parser.ts (pure) → Intent → dispatcher.ts (scope
     stack)
                      → screen scope | global scope | navigate + pending-intent
     replay
                      → tts.ts confirmations (holds the playback gate so
     detectors ignore it)

     - Two-tier commands: a global scope (navigation, metronome, tempo, stop,
     help, voice-off) registered once; screen scopes registered by each screen
     inside $effect(() => registerVoiceCommands({...})) (returns disposer), so
     voice mutates component-local $state through closures — no store refactor.
     - Cross-screen commands ("show me the D major scale" from Home): dispatcher
     stashes the intent (10 s TTL), navigate()s to the owning route, and
     replays it synchronously when the target screen registers its scope — no
     flash of default selection.
     - Grammar constraint: recognizer built with a word-list grammar generated
     by buildGrammar(lessons) (wake word, note letters, sharp/flat,
     scale/quality/inversion words, nav words, verbs, number words, lesson-title
     tokens, "[unk]"). Piano notes and stray speech decode as [unk] and are
     dropped before the wake-word check. Caveat: Kaldi silently drops
     out-of-vocab words — proper nouns like "Hanon" need speakable aliases
     ("finger exercise").
     - Wake word: "piano" (optionally "hey"/"okay" prefixed). Wake word alone →
     TTS "Yes?" + 8 s armed window where the next result parses without the
     prefix.
     - Self-hearing protection: audio chunks dropped while playback.isPlaying ||
     tts.speaking; TTS holds the playback gate so the pitch detectors also
     ignore speaker output; feedback strings never contain "piano".

     Edge cases

     - Mic denied → status='mic-denied', HUD shows mic.errorMessage, no retry
     loop. Model/WASM failure → status='error' + retry; only complete downloads
     are cached.
     - No recognizer restarts needed (unlike Web Speech) — Vosk runs while
     chunks flow; worker death caught via try/catch around acceptWaveformFloat →
     status='error'.
     - Enharmonics: chroma matching per target root list (Db in CHORD_ROOTS vs
     C# in minor roots).
     - BPM clamped 40–160 by consumers; wordsToNumber rejects outside 30–240.
     - Autoplay policy: suspended context yields silent chunks until first
     gesture — pointerdown unlock + HUD hint.
     - Two AudioWorkletNodes on one source is fine; addModule tracked per
     context (same pattern as polyPitch.svelte.ts).
