     Improve teaching quality, terminology tooltips, and user flows

     Context

     The app is content-rich but assumes musical literacy. Exploration found:
     - ~60 unglossed terms (arpeggio, inversion, cadence, half-diminished,
     ii–V–I, syncopation, …) appear as raw UI labels and in prose with no
     tooltip/glossary system anywhere. Scales/Chords libraries even render raw
     data IDs as button labels (ScalesLibrary.svelte:96 {t.id},
     ChordsLibrary.svelte:180 {q.id}).
     - Quiz feedback is binary — QuizCard.svelte:45-49 shows only "Correct!" /
     "It was X." with no why. The guide's excellent theory prose is siloed in
     guide.ts and never surfaces in the activities.
     - Flow dead-ends: guide deep-links have no return path; lesson/song
     completion offers only "again"; no first-run onboarding; good technique
     tips hidden in a collapsed <details>.
     - UI drift: .seg/.complete CSS duplicated across ~5 components; dead
     POLY_AVAILABLE lock code in Practice.svelte:10.

     Constraints: guide stays static, no progress tracking (established in a
     prior session); keep the UI simple (no libraries); pure logic stays
     vitest-testable; no new voice-facing vocabulary.

     Phase 1 — Glossary + tooltips (explicit ask)

     1. NEW src/lib/data/glossary.ts (pure):
       - GlossaryEntry { id, term, short, aliases? } — short is a 1–2 sentence
     beginner definition (≤ ~220 chars).
       - IDs for chord qualities / scale types must be exactly the
     ChordQualityId/ScaleTypeId union strings so libraries can look them up
     directly.
       - ~50 entries: intervals/semitone/octave, staff/clef/ledger line, key &
     time signature, tempo/BPM/bar, note values/dotted/syncopation/swing, all
     scale types + chord qualities, triad/inversion/arpeggio/broken chord,
     cadence kinds, tonic/subdominant/dominant, ii–V–I/guide tones/shell
     voicing/comping/walking bass, circle of fifths, relative minor, five-finger
     position, Hanon, contrary motion, chromatic, ABRSM grade, sight-reading,
     modes.
       - annotateGlossary(text): TextSegment[] — pure annotator:
     case-insensitive word-boundary match on term+aliases, longest-match-first,
     first occurrence only, cap ~4 terms per string.
     2. NEW src/components/Term.svelte (~70 lines): inline <button class="term">
     with dotted underline; CSS-only tooltip (position:absolute; bottom:100%,
     max-width 260px) shown on :hover/:focus-visible/click-toggle (Escape/blur
     closes); role="tooltip" + aria-describedby. Unknown term → plain text, dev
     warn. Self-contained styles.
       - Fix: remove overflow: hidden at GuideScreen.svelte:141 (would clip
     tooltips; safe — stage head has no bleeding background).
     3. NEW src/components/GlossText.svelte: {#each annotateGlossary(text)} →
     <Term> or plain text.
     4. Apply GlossText to prose spots: LessonPlayer.svelte:273 (description) +
     tips list, Practice.svelte:157 (METHOD_BLURBS), Quizzes.svelte:298 (mode
     desc), GuideScreen.svelte overview + theory body, Scales/Chords hint
     paragraphs. Do not wrap quiz answer options (leaks answers).
     5. Library labels: use lookupTerm(id).term (capitalized display name) for
     scale-type/chord-quality buttons; show lookupTerm(selectedId).short as a
     hint line in the card — defines "half-diminished" at the moment it's
     selected. No standalone /glossary page (keep simple).
     6. NEW src/tests/glossary.test.ts: every SCALE_TYPES/CHORD_QUALITIES id has
     an entry (import real registries, same pattern as guide.test.ts);
     annotator behavior (word boundaries, longest match, cap, round-trip); short
     length bounds.

     Phase 2 — Quiz "why" explanations (explicit ask)

     7. Add explanation: string to each option-based Question interface (echo
     mode skipped):
       - src/lib/ear/quiz.ts: exported INTERVAL_EXPLANATIONS (semitone count +
     song mnemonic), CHORD_QUALITY_EXPLANATIONS (recipe + character),
     SCALE_TYPE_EXPLANATIONS, CADENCE_EXPLANATIONS.
       - src/lib/theory/quiz.ts: reuse the ear tables (precedent: it already
     imports INTERVAL_LABELS); key-signature explanation computed via a small
     testable sigAccidentals() helper (sharp/flat order + "last sharp is a
     semitone below the key"); chord-function explanation computed ("in C major
     the 5th degree is G…"); note-naming uses per-clef mnemonics (Every Good Boy
     Does Fine / FACE) — cheap and still teaches.
       - src/lib/quiz/rhythmQuiz.ts: explanation = label + existing authored 
     hint from rhythms.ts.
     8. Display: optional explanation prop on QuizCard.svelte +
     RhythmQuizCard.svelte, rendered under the verdict after reveal (shown on
     correct answers too); pass it at the three call sites in Quizzes.svelte
     (~330, ~346, ~370).
     9. Tests: extend earQuiz.test.ts/theoryQuiz.test.ts (seeded-RNG pattern
     exists) — every question has non-empty explanation; interval explanation
     matches semitone distance; key-signature explanation lists the exact
     accidentals; test sigAccidentals directly.

     Phase 3 — User flows

     Browser back fix (URL-backed drill-downs). Today every screen reads
     currentParams() once at mount and holds selection in local $state
     (Practice.svelte:55-60,18, Songs.svelte:10-13, Quizzes.svelte:85-97), and
     nothing calls navigate() after mount. Consequences: opening a lesson/song
     pushes no history entry (back skips the list and leaves the screen
     entirely), and back/forward between param variants of the same route
     changes the URL but not the UI.

     9a. Router: add replace?: boolean to navigate() in src/router.svelte.ts
     (uses location.replace('#'+hash)) for URL updates that shouldn't create
     history entries.
     9b. Practice: drive catalog-lesson selection from the URL reactively —
     const params = $derived(currentParams()), selected lesson derived from
     params.lesson. Tile click → navigate('/practice', {lesson: id, 
     ...(params.from && {from: params.from})}); "← All lessons" / voice exit →
     navigate('/practice'). Sight-reading: "Sight-read at level N" →
     navigate('/practice', {sight: String(level)}) with the generated melody

     melody" regenerates in place with no navigation. Voice handlers switch from
     setting selected to calling navigate. {#key selected.id} already handles
     remount. Now: back from an open lesson returns to the lesson list; back
     from a guide-linked lesson returns to the guide.
     9c. Songs: same — catalog selection derived from params.song; tile click
     navigates; "← All songs" → navigate('/songs'). User-imported songs that
     aren't in the catalog keep local-state selection (not addressable by id).

     whenever they change, and derive them reactively from params —
     refresh/back-into-the-screen restores the mode without polluting history
     with every mode toggle.
     9e. Scales/Chords/Rhythm: leave once-at-mount reads (single-level screens;
     back already behaves). Optionally mirror the current selection via replace
     for refresh-safety — do it only if trivial.
     - Test: extend src/tests/routing.test.ts (or wherever buildHash/parseRoute

     Verification.

⏺ Done — and yes, it's a good idea in the hover-preview form, which is what I
  built. All 528 tests still pass, and 9 browser checks confirm the tooltips
  render and clicking still selects.

  How it works: a small CSS-only data-tip attribute style in app.css (visual
  twin of the prose tooltips) shows the definition on hover or keyboard focus,
  above the button. It's deliberately desktop-only: on touch there's no way to
  peek without selecting, so the definition line for the selected item (added
  last round) remains the touch path. Two input styles, both covered.

  Where it's now applied:
  - Chords — every quality button ("Augmented" → "A dreamlike, unsettled chord:
  two stacked major 3rds…"), the inversion buttons (root position vs. inversion
  definitions), and the Shell A/B voicing buttons.
  - Scales — every scale-type button.
  - Quizzes — mode buttons preview their description ("Cadences" → "A short
  progression plays — how does it end?") so you know what a mode is before
  entering it.
  - Rhythm Trainer — pattern buttons preview their counting hint ("Steady
  quarters" → "Tap right with the click…"), and the selected pattern's hint line
  now gets glossary terms too.
  - Songs — the "Grade 1/2/3…" headings explain the ABRSM grading scale on
  hover/tap.

  Where I deliberately didn't add them: quiz answer options — a tooltip defining
  "Half-diminished" on an option would hand you the answer. The post-reveal
  explanation covers learning there.

  One nuance to be aware of: hover tooltips on the quality buttons slightly
  duplicate the definition line shown for the selected quality. That's
  intentional redundancy — the hover is for comparing before you pick, the line
  is for whoever tapped straight in.
