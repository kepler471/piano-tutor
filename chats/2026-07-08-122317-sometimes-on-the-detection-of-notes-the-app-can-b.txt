‚ĚĮ Sometimes on the detection of notes the app can be quite slow to recognise.   
  For example if I play a scale but play it too fast even with the correct      
  keys, the app cannot keep up and so registers a missed key, meaning I have to 
  go back a few keys in the scale and retry it. Is there anything we can do to  
  improve this? And is this a limitation with the note detection model?         
                                                                                
  Some notes on this from a previous discussion are shown below:                
                                                                                
  Short answer: it's mostly not the model ‚ÄĒ it's the pipeline around it.        
  Pitchy's MPM algorithm returns a pitch for every frame in well under a        
  millisecond; the sluggishness comes from how frames are windowed, polled, and 
  confirmed. Here's the latency budget for a new note in the current setup      
  (monoTracker.ts + monoPitch.svelte.ts):                                       
                                                                                
  1. Analysis window fill (~40 ms+). The detector looks at the last 2048        
  samples (~43 ms at 48 kHz). A new note has to dominate that window before MPM 
  reports it ‚ÄĒ and while the previous scale note is still ringing (piano notes  
  decay slowly), the window contains two pitches, clarity drops below the 0.88  
  threshold, and frames get rejected.                                           
  2. Attack transient (~20‚Äď40 ms). A piano hammer strike is noise before it's   
  pitch. Those first frames also fail the clarity gate.                         
  3. Confirmation hysteresis (~33 ms minimum, often more). confirmFrames: 3     
  requires three consecutive valid detections of the same MIDI number, polled   
  at requestAnimationFrame rate (~16.7 ms on a 60 Hz display). Worse: a single  
  low-clarity frame in the middle (monoTracker.ts:105-109) resets               
  candidateCount to zero and you start over. On a 120 Hz display this is        
  faster; in a background-throttled or low-power tab, rAF can drop to 30 fps    
  and this alone becomes 100 ms.                                                
                                                                                
  Realistic total: 100‚Äď150 ms per note, sometimes worse. Sixteenth notes at 120 
  BPM are 125 ms apart ‚ÄĒ exactly where you're hitting the wall. And when a      
  note is missed, the matcher's "wait, don't fail" cursor stays put, so your    
  next correct notes all register as wrong.                                     
                                                                                
  The one part that is a genuine model limitation: MPM is strictly monophonic.  
  During the overlap where the old note sustains under the new one, it          
  fundamentally can't resolve two pitches ‚ÄĒ that's the irreducible chunk of     
  item 1. But everything else is tunable.                                       
                                                                                
  What I'd do, in order of impact-per-risk:                                     
                                                                                
  1. Decouple polling from rAF. Drive tick() from a short setInterval (~8‚Äď10    
  ms) or, better, from the capture AudioWorklet you already have for poly       
  detection ‚ÄĒ that halves the confirmation time and removes display-refresh/th  
  2. Tolerate one bad frame during confirmation. Instead of resetting           
  candidateCount on any invalid frame, allow a single miss before resetting.    
  This targethe worst-case path (a mid-confirmation glitch doubling the         
  latetave-flicker protection much.                                             
  3. Make the hysteresis interval-aware. The 3-frame confirm exists to kill     
  octave flicker ‚ÄĒ but octave errors are specifically ¬Ī12/¬Ī19 semitones from    
  the ringing note. For a candidate that's a step or two away (i.e., t is safe; 
  keep 3 only for octave-related jumps.                                         
  4. Matcher-level lookahead. In StepMatcher, if an onset matches the next step 
  while the current one is incomplete, treat the current step as                
  done-but-stumbinstead of flagging a wrong note. This makes a single missed    
  detst runs ‚ÄĒ arguably the biggest UX win regardless of detectiontuning.       
                                                                                
                                                                                
  Items 1‚Äď3 have a safety net: the synthesized-audio tests in                   
  src/tests/detection.test.ts can be extended with a "fast scale" fixture (e.g. 
  8 notes/sec with realistic decay overlap) so we can verify no note is         
  dropped befthe project convention.                                            
                                                                                
  Want me to implement these? I'd start with the fast-scale test f then apply   
  1‚Äď3, and treat 4 as a separate decision since itchanges practice-mode         
  semantics.

     Faster note detection for quick scales

     Context

     Playing a scale quickly causes missed notes: mono detection latency is
     ~75‚Äď150 ms/note, and once one note is dropped the StepMatcher cursor stays
     put, so every following correct note flashes "wrong" and the user must back
     up and retry. The lag is mostly pipeline, not the pitch model:

     - tick() in src/lib/audio/monoPitch.svelte.ts:47-62 is driven by
     requestAnimationFrame (~16.7 ms at 60 Hz, worse when throttled), and
     MonoTracker needs confirmFrames: 3 consecutive valid frames ‚Üí ‚Č•50 ms
     confirmation on top of window fill and the attack transient.
     - A single invalid frame mid-confirmation
     (src/lib/audio/monoTracker.ts:105-110) zeroes candidateCount, restarting
     the count.
     - The 3-frame confirm exists to suppress octave flicker, but octave errors
     are specifically octave-related intervals (¬Ī12/¬Ī19/¬Ī24) from the ringing
     note ‚ÄĒ stepwise scale motion doesn't need the full 3 frames.
     - StepMatcher (src/lib/practice/matcher.ts:48-66) has no lookahead, so one
     dropped detection derails the rest of the run.

     Note: the capture AudioWorklet posts non-overlapping 2048-sample chunks
     (~46 ms cadence) ‚ÄĒ slower than rAF ‚ÄĒ so it is NOT the right polling source;
     a short setInterval reading the AnalyserNode is.

     User decision: include the StepMatcher lookahead alongside the detection
     fixes.

     Expected latency after: ~23 ms window fill + 2‚Äď3 √ó 10 ms confirm ‚Čą 43‚Äď53 ms
     per note (vs 75‚Äď150 ms today), and a genuinely dropped detection no longer
     cascades into a wall of wrong-note flashes.
