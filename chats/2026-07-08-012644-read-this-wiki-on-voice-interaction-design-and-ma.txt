     Voice UI improvements from the ACIxD wiki (acixd.org)

     Context

     The user asked me to read the ACIxD voice-interaction design wiki
     (Association for Conversational Interaction Design — research-backed VUI
     practice) and apply it to the app's voice interface. The wiki's guidance on
     error recovery, confirmation strategies, universal commands,
     conversational maxims, and grammar/mimicry maps onto concrete gaps in our
     voice stack:

     - Every recognition miss gets the same fixed "Sorry, I didn't catch that…"
     — the wiki prescribes escalating reprompts ("rapid reprompt" → context
     examples → switch modality, which for us means pointing at the visual HUD
     list).
     - The embedding fallback is binary accept/reject — the wiki prescribes
     explicit confirmation on low confidence ("…, right?") and a distinct
     disconfirmation response ("My mistake.", not a no-match message).
     - Missing the wiki's universal commands repeat and go back.
     - No one-step correction ("no, D minor") — today you must re-say the wake
     word after a misrecognition.
     - Spoken help lists 4 examples (wiki: >2 "sounds like a menu"); the
     recognizer is deaf while TTS speaks, so all feedback must stay terse.
     - The wiki's mimicry principle (users parrot what the app advertises) is
     untested — and indeed Quizzes advertises "hear it again", which the regex
     parser cannot parse (works only via embedding fallback). An invariant test
     will lock this down.

     User confirmed all four improvement areas: error recovery & correction,
     universal commands, prompt wording, grammar/mimicry test + miss log.

     Architecture decision

     A/B/D (reprompts, did-you-mean, correction window) all mutate "what happens
     after an utterance" state. They become one pure state machine
     src/lib/voice/convo.ts (injected-env pattern like dispatcher.ts,
     vitest-covered without a browser). Modes: idle | armed | confirming, with
     until timestamp (lazily expired — no timers), misses counter, suggestion: 
     Intent | null. voice.svelte.ts keeps async (fallback embedding) + DOM (tts,
     history) and delegates decisions to the convo. Dispatcher stays pure; it
     loses only its unknown-branch messaging.

     Key transition rules:
     - onWake → armed 8s, say "Yes?" (not remembered for "repeat").
     - onIntent handled → misses=0, arm 5s correction window (wiki one-step
     correction).
     - onFallback(null) → misses++, escalating reprompt: 1→"Sorry?"; 2→two
     examples from the top (active) scope's phrases; ≥3→Tap "What can I say" for
     the full list.
     - onFallback(suggest) → confirming 8s, say ${describeIntent(intent)} — 
     right?.
     - affirm while confirming → dispatch suggestion; deny bare → "My mistake."
     (misses NOT incremented — wiki: disconfirmation ≠ no-match); deny with
     correction → dispatch correction; timeout → drop silently; any other full
     command cancels the suggestion and dispatches normally.
     - Piano-audio false-positive exposure is acceptable: grammar decodes notes
     as [unk], recognizer is deaf during playback/TTS; a stray bare "no" costs
     only "My mistake." Document in convo.ts.

     2. Universals: "repeat" + "go back"

     - tts.ts: remember lastSpoken; speak(text, {remember?}) — meta-prompts
     ("Sorry?", "Yes?", "My mistake.", confirmations) pass remember:false so
     "repeat" never replays them.

     remains the lesson command). Add repeat did to grammar staticWords.
     - voice.svelte.ts global scope: repeat → say tts.lastSpoken || "I haven't 
     said anything yet."; go-back → history.back() (back UX is URL-backed). Do
     NOT shadow go-back in LessonPlayer — universals stay consistent (wiki).
     - Extend src/tests/voiceParser.test.ts.

     3. "Voice is on." enable confirmation

     - voice.enable({announce?: boolean}): speak 'Voice is on.' (remember:false)
     once listening starts. VoiceButton toggle + HUD Retry pass announce:true;
     the auto-restore path on page load stays silent (also sidesteps
     gesture-gated TTS).

     4. Convo state machine + escalating reprompts

     - dispatcher.ts: delete unknown-branch messaging (lines 79-83) — returns
     {handled:false, feedback:''}; messaging moves to convo. Update
     voiceDispatcher.test.ts.
     - New src/lib/voice/convo.ts (createConvo, repromptFor,
     ARMED_WINDOW_MS=8000, CORRECTION_WINDOW_MS=5000, CONFIRM_WINDOW_MS=8000).
     - voice.svelte.ts: delete armedUntil; handleFinal becomes thin routing into
     convo; parseTranscript(text, {armed: convo.isArmed()}).
     - New src/tests/voiceConvo.test.ts (env-spy style with injected now):
     escalation sequence, counter reset on success only, window expiry,
     remember:false on meta prompts, no "piano" in any reprompt.

     5. Correction window + "no," parsing

     - intents.ts: {kind:'deny', correction?: Intent}; parser.ts: extract
     internal parseCommand(tokens) (pure refactor), then leading no → deny with
     parsed correction; add no to staticWords explicitly.
     - convo: deny handling per transition table; voice.svelte.ts routes
     deny-with-unknown-correction text through the fallback path (reuses
     fallbackGate).
     - Tests: parser ("piano no d minor"), convo (correction within window; bare
     "no" → "My mistake."; idle "no" silent).

     6. Confidence band + "did you mean" (highest-risk, last)

     - intentMatcher.ts: SUGGEST_THRESHOLD = 0.45 + matchIntentBanded() → accept
     (current rules) | suggest (fails accept but score ≥ 0.45 — covers both
     margin-failure and near-miss-below-threshold). matchIntent becomes a
     wrapper returning only accepts (existing tests/callers stay green).
     - fallback.ts: FallbackOutcome = {kind:'match'|'suggest', intent} | null;
     suggest whose resolve() returns null degrades to null (never confirm an
     unexecutable intent).
     - New src/lib/voice/describe.ts: describeIntent(intent) short spoken
     paraphrase (move SCREEN_NAME_FOR_ROUTE from dispatcher.ts:23 into
     intents.ts, re-import). Invariants: never "piano", never empty for
     bank-producible kinds. New voiceDescribe.test.ts.
     - parser.ts: {kind:'affirm'} for yes|yeah|yep (NOT "right" — collides with
     right hand). Grammar += yes yeah yep (total growth across all steps ~4
     words — negligible for Vosk).
     - Calibration: split the integration test's negatives into HARD_NEGATIVES
     (must be fully null, below 0.45) and NEAR_MISS_ROWS (must not be 'match';
     'suggest' acceptable). Tune 0.45 from the logged run —
     src/tests/voiceEmbedding.integration.test.ts remains the calibration
     authority (never hand-tune).

     7. Wording pass (anytime after step 1)

     - Help (voice.svelte.ts:122): rebuild from SPOKEN_HELP_EXAMPLES — 2
     examples + For everything else, tap "What can I say". (≤2 examples,
     contractions, new-info-last).
     - GuideScreen.svelte:52 → "Stages go from one to five." (affirmative);
     FreePlay.svelte:103 → "First say 'start listening', then 'record'."
     (instruction order = action order).
     - Keep already-conformant strings ("Opening Scales.", "Left hand.", "I
     can't do that on this screen.").

     8. Miss log (optional tuning aid)

     - New src/lib/voice/missLog.ts: capped (~50) localStorage ring buffer
     piano-tutor.voice-misses of unrecognized transcripts, injected as
     ConvoEnv.logMiss. No UI; read via devtools to tune grammar/intent bank from
     real data (the wiki's tune-from-production loop).

     Verification

     - npm test — new voicePhrases/voiceConvo/voiceDescribe tests + extended
     parser/dispatcher/matcher/fallback tests; npm run check.
     - npx vitest run src/tests/voiceEmbedding.integration.test.ts — recalibrate
     SUGGEST_THRESHOLD from its logged scores.
     - Manual E2E (npm run dev): enable voice → hear "Voice is on."; say
     gibberish 3× → "Sorry?" → examples → HUD pointer; "piano, show me D major"
     then "no, D minor" within 5s → corrects; borderline phrasing (e.g. "piano,
     let me see the D chord please") → "… — right?" → "yes" executes / "no" →
     "My mistake."; "piano, repeat" replays last real feedback; "piano, go back"
     steps back; Quizzes "piano, hear it again" now works via the parser.

     Risks

     - Calibration churn in step 6: if hard negatives score above 0.45, raise
     SUGGEST_THRESHOLD or accept occasional harmless "— right?" prompts (a "no"
     costs one turn). Decide from the integration test's logged run.
     - Behavior change: "go back" inside a lesson now means browser-back (was:
     previous segment; "previous part" still does that).
     - Step 1's invariant test may surface more broken advertised phrases beyond
     "hear it again" — fix parser or phrase case by case.

  What the wiki changed in the app:

  1. Escalating reprompts (wiki: "stepping stones and safety nets"): instead of
  the same "Sorry, I didn't catch that…" every time, a miss now gets "Sorry?", a
  second miss gets two example phrases from the current screen's commands ("You
  can say: show me d major, or left hand."), and a third points at the visual
  HUD list and goes quiet until re-woken — the multimodal equivalent of an IVR's
  touchtone fallback.
  2. "Did you mean … — right?" confirmation: the embedding fallback was binary
  accept/reject; there's now an uncertain band (SUGGEST_THRESHOLD 0.45 ≤ score <
  accept gates) that asks aloud instead of failing. Recalibrated against the
  real MiniLM model: hard negatives peak at 0.440 (never prompt), on-domain
  chatter peaks at 0.478 (worst case is one cheap "no"), true paraphrases stay
  above 0.55 (still act silently).
  3. One-step correction (wiki: allow "No, $500"): every successful command
  opens a 5-second window where "no, d minor" corrects without repeating the
  wake word, and bare "no" gets "My mistake." — deliberately distinct from a
  no-match message, per the wiki's disconfirmation guidance.
  4. Universal commands: added "repeat" / "say that again" (re-speaks the last
  real feedback; meta-prompts like "Sorry?" are excluded) and "go back" (browser
  back — your URL-backed navigation made this free). "Go back" inside a lesson
  now means back-navigation; "previous part" still steps segments.
  5. Prompt wording per the conversational maxims: spoken help cut from 4
  examples to 2 (wiki: three "sounds like a menu") plus a pointer to the HUD;
  "There are five stages." → "Stages go from one to five." (affirmative — tells
  the user what to say); enabling voice now confirms "Voice is on."
  (user-initiated enables only, silent on page reload).
  6. Mimicry invariant test (wiki: users parrot what you advertise): all
  advertised phrases moved to a single registry (src/lib/voice/phrases.ts) and a
  new test enforces that every one parses and survives the Vosk grammar. It
  immediately caught two real bugs — "hear it again" (advertised on Quizzes,
