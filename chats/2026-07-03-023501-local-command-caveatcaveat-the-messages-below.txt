     Piano Tutor — Browser Piano Learning App

     Context

     Greenfield app in the empty directory /Users/steli/src/piano-tutor. The
     user is a beginner pianist with an electric piano that has no MIDI 
     connection, so the app must listen through the microphone. Requirements:

     - Show all scales and lots of chords — each with sheet music and correct 
     fingering
     - Listen to playing, show pressed keys on an on-screen keyboard in real
     time, and transcribe to sheet music
     - Include well-known beginner practice methods, each with sheet music +
     fingering + play-along feedback

     Decided with the user: Web app, Svelte 5 + TypeScript + Vite, dual
     detection — fast monophonic DSP (instant feedback for scales/melodies) plus
     polyphonic ML (Spotify Basic Pitch) for chords and transcription.

     Stack & key decisions

     Concern: Notation
     Choice: vexflow@^5
     Why: Programmatic score building, per-note SVG highlighting, first-class
       Fingering modifier. All usage isolated in src/lib/notation/ so a fallback
     pin
        to 4.2.x touches one folder.
     ────────────────────────────────────────
     Concern: Theory
     Choice: tonal@^6
     Why: Computes all scale/chord spellings, MIDI↔name, enharmonics. Fingerings
     are
        hand-authored (pedagogical conventions, not derivable).
     ────────────────────────────────────────
     Concern: Mono pitch
     Choice: pitchy@^4 (MPM autocorrelation)
     Why: Sub-ms detection; MVP feeds it from an AnalyserNode, later an
       AudioWorklet.
     ────────────────────────────────────────
     Concern: Poly pitch
     Choice: @spotify/basic-pitch@^1 in a Web Worker on the tfjs WASM backend
     Why: WebGL is flaky in workers; WASM is worker-safe. Sliding-window
     inference
       (2 s window every 500 ms) since Basic Pitch is offline-only.
     ────────────────────────────────────────
     Concern: Playback
     Choice: tone@^15 Sampler + Salamander piano samples
     Why: Hear example scales/chords as piano, plus metronome.
     ────────────────────────────────────────
     Concern: Routing
     Choice: Hand-rolled hash router (~30 lines)
     Why: 6 screens; no server config needed on static hosting.
     ────────────────────────────────────────
     Concern: Threading
     Choice: postMessage + transferable Float32Arrays, no SharedArrayBuffer
     Why: Avoids COOP/COEP headers; trivial static hosting.

     ⚠️ @spotify/basic-pitch pins @tensorflow/tfjs@^3.x — align the tfjs install
     with what it resolves (npm ls @tensorflow/tfjs) to avoid bundling two TF
     runtimes.

     Scaffolding

     npm create vite@latest . -- --template svelte-ts
     npm i vexflow tonal pitchy @spotify/basic-pitch @tensorflow/tfjs
     @tensorflow/tfjs-backend-wasm tone
     npm i -D vitest jsdom

     Copy the Basic Pitch model from node_modules/@spotify/basic-pitch/model/ →
     public/model/ so the worker fetches it same-origin.

     Detection design (the hard part)

     - Mono path (scales, melodies, tuner): AnalyserNode → pitchy →
     hysteresis/median filter → NoteEvent. ~50–120 ms perceived latency; honest
     about it.
     - Poly path (chords, transcription): stream 22050 Hz chunks to worker;
     every 500 ms run evaluateModel() on the last 2 s;
     outputToNotesPoly(onsetThresh≈0.5, frameThresh≈0.3, minLen≈5); emit only
     notes not already emitted (same MIDI within ±120 ms). 0.25–1 s behind real
     time — fine for chord checking.
     - Coexistence: explicit per-activity detectionMode: 'mono' | 'poly' — no
     fusion in v1. Keyboard highlight may show the union; grading uses only the
     designated source.
     - Speaker/mic loop: session.ts enforces demo-mode vs listen-mode
     exclusivity; pause detection during playback + 300 ms tail.

     Transcription (pragmatic, no rhythm detection in v1)

     - Tier A (Free Play default): every onset → quarter note; onsets within 80
     ms cluster → chord. No rests/durations. Ships first.
     - Tier B (record with metronome): user picks tempo + time signature;
     count-in; onsets snap to 8th-note grid; gaps become rests; bars split by
     time signature.

     Practice content (v1, beginner)

     1. Five-finger positions (C, G, F) — mono
     2. Hanon No. 1 (first 8 bars, one octave, slow) — mono
     3. Scale routine: C/G/F major, 1 then 2 octaves — mono
     4. Cadence drills: I–IV–V–I block chords in C/G/F — poly
     5. Sight-reading: generated random 4-bar diatonic melodies in C position —
     mono

     Matcher is beginner-friendly: next expected note highlighted on staff +
     keyboard; correct onset advances cursor (green), wrong note flashes red but
     waits. Chord lessons compare collected poly onsets as note sets, showing
     missing/extra chord members. Grade on onsets only (sustain pedal makes
     offsets unreliable; lessons say "no pedal").

     Screens

     #/ Home · #/scales ScalesLibrary (picker → notation + fingering + keyboard
     + play + "Practice this") · #/chords ChordsLibrary (root × quality ×
     inversion → same + "Check my chord") · #/practice LessonPlayer · #/play
     FreePlay (live keys + rolling transcription + record) · #/tuner Tuner (note
     + cents; doubles as detection debug) · #/dev fixture test bench (DEV only)

     Build order (each phase independently testable)

     - Phase 0 — Scaffold: Vite/Svelte/TS, router, theory/ on tonal, clickable
     PianoKeyboard, vitest. ✓ unit tests pass, keys click.
     - Phase 1 — Reference tool: vexScore, SheetMusic, fingering data, Tone
     playback, Scales + Chords libraries. ✓ browse all 36 scales / chord grid
     with notation + fingering, hear them. App already useful with no mic.
     - Phase 2 — Mono detection: mic.ts, monoPitch, Tuner, live keyboard
     highlight, FreePlay Tier A, fixture test bench. ✓ play single notes, keys
     light up, notes appear on staff.
     - Phase 3 — Practice engine: matcher, session, LessonPlayer, Metronome,
     lessons 1/2/3/5. ✓ complete a five-finger lesson with green/red feedback.
     Product milestone.
     - Phase 4 — Polyphony: Basic Pitch worker + WASM, AudioWorklet capture,
     chord check, cadence lessons. ✓ play C-E-G, app names the chord.
     - Phase 5 — Transcription Tier B + polish: metronome-locked recording,
     quantize grid, rests/bars, localStorage practice history, A4 calibration
     setting.

     Risks & mitigations

     - Piano harmonics/octave errors: mono — clarity threshold + 3-hop
     hysteresis + median filter; poly — tune thresholds against fixtures,
     post-filter notes that are exact octave/12th of a louder simultaneous note.
     - Basic Pitch load (~4–5 MB with tfjs-wasm): lazy-load worker only when a
     poly feature opens; loading state; warm-up on silence.
     - Safari: AudioContext requires user gesture — gate behind "Start
     listening" button; 22050 Hz context fallback via resample.ts.
     - VexFlow 5 doc gaps: all usage in notation/; 4.2.x is a drop-in fallback.

     Verification

     - Fixture audio bench (primary loop, no piano needed): WAVs in
     public/fixtures/ (synthesized tones + real recordings of single notes, C
     major scale at 60/120 bpm, C/G/F triads, I-IV-V-I). Dev route #/dev decodes
     and pumps chunks through the exact same detector code paths; displays
     detected vs expected with timing error. Also how thresholds get tuned
     reproducibly.
     - Unit tests: theory (scale/chord correctness), fingerings (every library
     item has an entry; RH C major === [1,2,3,1,2,3,4,5]), quantize (synthetic
     events → expected grid), matcher (cursor behavior).
     - Manual checklist: mic-denied path; Safari/Firefox smoke; all 36 scales
     render with correct accidentals + fingering; tuner A4=440 ±5 cents vs phone
     tuner; five-finger lesson completable; chord check rejects missing 3rd;
     slow C-scale transcription at 60 bpm yields 8 correct quarters.

  - Transcription — Free Play writes everything you play onto a staff and names
  chords it recognizes. "Record with metronome" gives you a count-in, then
  quantizes your playing to a real rhythm — quarter/eighth notes, rests, and
  barlines.
  - Practice methods — five-finger positions, Hanon No. 1, a scale routine,
  I–IV–V–I cadence drills, and randomly generated sight-reading melodies. The
  lesson player highlights the next note in orange, shows the target key and
  finger in green, flashes wrong notes red without failing you, plays demos, has
  a metronome, and logs completed segments to a practice history on the Home
  screen.
  - Note Detector — a tuner-style screen showing the note, frequency, and cents
  offset, useful for checking what the app hears.

  How it was verified

  Detection was tested by synthesizing piano-like audio (single notes, a full C
  major scale, C-E-G chords) and running it through the exact code paths the mic
  uses — the mono tracker correctly detected the scale in order, and Basic
  Pitch found all three chord tones. That test also exposed harmonic "ghost"
  notes the model reports an octave above loud notes, so the worker now filters
  those. I then drove every screen in headless Chrome and inspected screenshots.

  Two things I couldn't test without your piano: real-microphone accuracy (the
  detection thresholds may need tuning against your room and instrument — the
  Note Detector screen is the debug view for that) and the piano-sample
  playback, which streams from tonejs.github.io on first use, so it needs
  internet.

  One deviation from the plan: instead of fixture WAV files and a #/dev bench,
  detection is verified with synthesized audio in the unit tests, which is
  reproducible and runs in CI.
