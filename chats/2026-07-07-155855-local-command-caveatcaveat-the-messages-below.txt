⏺ Yes — once per page load. Here's the lifecycle:

  - startPolyDetection() creates the worker only if it doesn't already exist (if
  (!worker) at polyPitch.svelte.ts:74), and stopPolyDetection() (line 99) only
  disconnects the audio worklet — it never terminates the worker. So toggling
  chord detection off and on, or switching screens, reuses the already-loaded
  model.
  - Inside the worker, init() (basicPitch.worker.ts:39) runs once: sets up the
  TF.js WASM backend, loads the model from /model/model.json, and does a silence
  warm-up inference so kernel compilation doesn't happen on your first real
  chord.

  Across page reloads it does re-initialize (new worker, new WASM backend,
  warm-up again), but the model files themselves are static assets so the
  browser's HTTP cache typically spares the re-download — the reload cost is
  mostly the kernel compile/warm-up, not network.

  So within a session: first activation pays the loading cost, everything after
  is instant.

⏺ You were right — it wasn't loading, it was crashing. Fixed now.

  The bug: Basic Pitch frames audio with tf.signal.frame(..., padEnd: true), and
  tfjs-core's padding calls fill() without a dtype. The CPU backend tolerates
  that, but the WASM backend's Fill kernel throws Unknown dtype undefined. Since
  every 2-second window needs end-padding, the very first inference (the
  warm-up on silence) died instantly — in dev and in the production build. This
  is a bug in the pinned tfjs 3.21, which we can't upgrade because basic-pitch
  pins it.

  The fix (in src/lib/audio/basicPitch.worker.ts): after the backend is ready,
  re-register the WASM Fill kernel with a thin wrapper that defaults the missing
  dtype to float32. It's guarded by getKernel('Fill', 'wasm'), so the CPU
  fallback path is untouched.

  Verified end-to-end with headless Chromium driving the real worker: it now
  reports ready on the wasm backend in ~450 ms, and a synthesized C-major chord
  streamed through the chunk protocol comes back as midi 60/64/67 — in both the
  dev server and the built dist/. npm run check is clean and all 494 tests pass.
